## 3. LANDMINES — read before you touch the filesystem or git

This is a synced/mounted Windows drive seen through a Linux sandbox. It misbehaves in specific, repeatable ways. The originally-handed-off doc warned about file staleness (its §3.4); this session that warning turned out to be an understatement.

### 3.1 File-sync truncation is real and it silently committed a broken file.
Files edited via the editor/Write tool read back **truncated** from the shell (`cat`/`grep`/`wc`/`python import`) — cut off mid-line, missing the back half — while the canonical editor view is correct. This session it hit `config.py`, `service.py`, and worst of all **`mcp_server.py` at `git add` time**: git hashed a truncated working-tree read and committed a blob with an unterminated docstring (lost its last 5 lines). Tests didn't catch it because CI didn't (yet) install `mcp`. It was caught only by parsing the *committed blob*.

**Mitigations that worked (use them):**
- After any non-trivial edit, **rewrite the whole file from the shell with a quoted heredoc** (`cat > f <<'EOF' … EOF`) using content you've freshly confirmed, then verify **in the same shell call** with `wc -l` + `python3 -c "import ast; ast.parse(open('f').read())"`.
- When committing, **stage risky files via `git hash-object -w /tmp/copy`** (a sandbox-local, non-mount file) + `git update-index --cacheinfo`, rather than `git add` on the mount path — this bypasses the flaky mount read entirely.
- **Always verify the committed blob**, not just the working tree: `git show HEAD:path | python3 -c "import sys,ast; ast.parse(sys.stdin.read())"`. Do this for every `.py` in every commit here.

### 3.2 There's a stuck `.git/index.lock` you cannot delete.
A zero-byte `.git/index.lock` dated Jun 30 (a crashed process) survives `rm` even after deletes are enabled — the mount won't unlink that specific inode. So a normal `git commit` fails with "index.lock exists." **Work around it by committing through a temp index:** `export GIT_INDEX_FILE=/tmp/idx; git read-tree HEAD; git add/update-index …; tree=$(git write-tree); c=$(git commit-tree $tree -p HEAD -F msg); git update-ref refs/heads/main $c; cp /tmp/idx .git/index` (the final `cp` syncs the canonical index so `git status` is clean). Every commit this session used this pattern.

### 3.3 The mounted folder blocks file deletion by default.
`rm` returns "Operation not permitted" in the user folder until you call the `mcp__cowork__allow_cowork_file_delete` tool (approves normal deletes). Even after that, the stuck lock in 3.2 stays stuck. `touch`/write always work; it's only unlink/rename that were gated.

### 3.4 No push credentials in the sandbox.
`git push` fails ("could not read Username"). The human must push from their machine. Read access to `origin` works (public repo), so `git ls-remote` / reading the GitHub Actions page is fine for verification — but it was intermittently empty; the browser Actions page was the reliable check.

### 3.5 Another session is editing this repo concurrently.
Around 06:32 today a **separate** session added `skills/engraphis-memory/` (a portable Agent Skill), `.claude-plugin/` (`marketplace.json` + `plugin.json`), and edited `AGENTS.md`/`README.md` to reference the skill. These are **uncommitted in the working tree** and were deliberately left untouched — none of my commits stage them. Before you commit broadly (e.g. `git add -A`), decide what to do with that work, and be aware `main` may have another writer.

## 5. Before you believe any status claim (including this one)

Re-run the gate yourself. With the new `test` extra this is one install:

```bash
pip install -e ".[test]"          # full offline stack, no torch
python -m pytest tests/ -q         # expect 139 passed
python -m eval.harness --dataset eval/datasets/sample.jsonl --k 5
python -m eval.harness --dataset eval/datasets/codemem.jsonl --k 5
```

And when you fix something non-trivial, reproduce the original bug independently (a standalone script / fresh payload) and confirm the fix holds outside the unit test you wrote — that's what caught both the original bugs and, this session, the `mcp_server.py` truncation and the read-isolation leak.
